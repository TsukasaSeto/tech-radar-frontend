## プロンプト改善提案（4件）

### 1. [中] TypeScript ルール群の「バージョン感度」チェックの明示 [v1]
- **対象セクション**: 採用・修正基準
- **現状の記述**: 採用基準はあるが「既存ルールが最新バージョンと矛盾していないか」の積極的チェックについて明示がない
- **提案する変更**: 強化/修正フェーズで、対象ルールのコード例・バージョン注記と記事の主張を突き合わせ、「既存推奨パターンが非推奨化されていないか」を確認するステップを追加する
- **理由**: 今回 `baseUrl` のように、かつての推奨パターンがメジャーバージョンで非推奨化される場合、記事がなければ発覚しないまま古い推奨が残り続ける。「TypeScript N.0」「Next.js N.0」などのメジャーリリース記事は積極的に修正候補としてチェックすべき
- **想定リスク**: チェック対象が広がりすぎると時間超過の懸念あり
- **優先度**: 中

### 2. [低] 継続的失敗ソースの「スキップ開始段階」テーブルを更新（Turborepo / Mercari）[v1・v2 統合]
- **対象セクション**: 「段階1 をスキップするソース（既知の継続的失敗）」テーブル / `標準RSSソース定義 > 公式ブログ・本家`
- **現状の記述**: Netflix Tech Blog のみが段階2（rss2json）始まりとして指定されている
- **提案する変更**: Turborepo Blog（403 継続）と Mercari（空レスポンス継続）を、連続失敗が確認された場合に Netflix と同様のフォールバック指定（段階2 始まり）としてテーブルへ追記する
- **理由**: 2026-06-06 でも同様の 403 が発生しており継続的。段階1 で繰り返し失敗するソースを毎回試みるのは時間の無駄。3回連続失敗を閾値に既知失敗リストへ昇格させるルールは既にあるが、Mercari の「空レスポンス」が "失敗" とカウントされるかが曖昧
- **想定リスク**: フォールバック試行数が増える（微小）/ 一時的障害と恒常的失敗の区別が難しい
- **優先度**: 低

### 3. [低] 「採用却下→翌日再試行」のパターン追記 [v2]
- **対象セクション**: 採用基準 Pattern 2 の「直近2ヶ月以内」条件
- **現状の記述**: 「同じ知見が2記事以上で言及されている（直近2ヶ月以内）」
- **提案する変更**: 「今回 D: 却下した記事の URL を read_skipped に記録する際、記事の主題（例: 'AI前提E2Eテスト再設計'）を changelog の却下理由欄にも記録する。翌日同じ主題の記事が出た場合に Pattern 2 として採用できる」という運用ヒントを追記
- **理由**: 今回 sompojapan_dx の E2E 記事や env variable 記事が「単一記事で Pattern 2 未達」で却下されたが、翌日類似記事が出た際に今回の却下記録が活用できると効率的
- **想定リスク**: 実装コスト（changelog grep が複雑化）
- **優先度**: 低

### 4. [低] ソース網羅性（多タグ・Medium）の時間内消化 [v2]
- **対象セクション**: Step 4 ソース定義 / 巡回順序
- **現状の記述**: 24ソース定義だが、Qiita・dev.to・Medium の多タグを毎回消化しきれていない

- **理由**: 毎回ソース数が多く時間内に全タグを確認しきれず、採用漏れの可能性が残る
- **想定リスク**: 巡回対象を絞ると高品質記事を取りこぼすリスク
- **優先度**: 低
