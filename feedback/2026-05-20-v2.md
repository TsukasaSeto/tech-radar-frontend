## 観察した問題（1件）

### 1. 同日 v1 Routine が D 判定した記事を v2 が A 採用 — 採用基準の一貫性問題
- **種類**: 採用判定の一貫性
- **発生箇所**: Step 5 / 採用前チェック → 採用判定
- **詳細**: 本日の v1 Routine（PR #69）は React Compiler annotation mode 記事を「Pattern 2 未達（単一コミュニティ記事）」として D 判定した。v2 Routine（本実行）では同記事を「Pattern 1（公式 React Compiler ドキュメントで明記された機能の実装ガイド）」として A 採用した。
  - v1 の D 判定: コミュニティ記事単体では Pattern 2（複数独立ソース）を満たさないと判定
  - v2 の A 採用: `compilationMode: "annotation"` は公式ドキュメント記載の機能であり、コミュニティ記事はその実プロジェクト適用例として Pattern 1 に該当すると再評価
- **影響**: 同日内で判定が逆転した。後続 Routine（v3 以降）が再び同記事を評価した場合、`_seen.json` に登録済みであるため二重採用は防止される
- **再現性**: 「公式ドキュメントが存在するが、採用根拠はコミュニティ記事」という境界ケースで発生しやすい
- **推奨対処**: プロンプトに「公式ドキュメントで明記された機能の場合、コミュニティ記事は Pattern 1 の実装ガイドとして扱い、単独でも採用可」という明示的な規定を追加する

## プロンプト改善提案（1件）

### 1. Pattern 1 の適用範囲を「公式機能 + コミュニティ実装例」に明示する
- **対象セクション**: 採用基準 / Pattern 1 の定義
- **現状の記述**: "Pattern 1: 公式ドキュメント・仕様に基づく内容（1ソースでも採用可）"
- **提案する変更**: "Pattern 1: 公式ドキュメント・仕様に基づく内容。**公式ドキュメントが参照先として存在し、コミュニティ記事がその実装手順・適用例を提供している場合も Pattern 1 に含む**（1ソースでも採用可）" と明示する
- **理由**: v1 が D、v2 が A と判定が割れたのは、「公式 API を使っているが掲載媒体はコミュニティ記事」という灰色ゾーンの扱いが未規定だったため。明示することで Routine 間の一貫性が向上する
- **想定リスク**: Pattern 1 の適用範囲が拡大し、採用件数が増える可能性がある（品質への影響は軽微と予想）
- **優先度**: 高
